# VerbaLab 已验证经验

## 如何阅读这些经验

这不是 bug 日志，也不是完整编年史。这里只保留那些已经被反复证明对后续改动有指导意义的经验：为什么会出问题、什么修法有效、下次遇到类似情况应该直接复用什么模式。

## 已验证有效的模式

### 1. 不要让页面临时状态冒充已持久化数据

**问题**  
页面看起来显示对了，但刷新后又丢，下一次还要重新生成或重新拉取。

**为什么会发生**  
补全结果只写到了页面内部状态，没有进入 store、本地存储和同步链路。

**有效修法**  
把结果写回正式数据模型，再由 store 持久化与同步。

**下次复用方式**  
凡是学习资产类结果，只要用户下一次还会用到，就默认先问一句：它是不是已经进了 store，而不只是进了组件 state。

### 2. 不要用静默 fallback 伪装 AI 成功

**问题**  
AI 失败后，页面展示一段“看起来像建议”的通用内容，但和用户输入几乎无关。

**为什么会发生**  
调用方把失败隐藏掉，用本地通用文案顶上，看起来像成功返回。

**有效修法**  
把“允许 fallback”变成显式选项。对必须高度相关的场景，失败就直接报错，不伪装成功。

**下次复用方式**  
先分清这个页面需要的是“有内容就行的兜底”，还是“必须和当前输入强相关的 AI 结果”。后者不要静默 fallback。

### 3. 关键表达不要从长文案里猜

**问题**  
卡壳点、词卡、语体解析等区域，本应显示一个短表达，却时而抓到错误句，时而抓不到任何内容。

**为什么会发生**  
系统试图从一大段自由文本里反向提取关键字段，格式稍微漂一点就失稳。

**有效修法**  
让 AI 直接返回结构化字段，例如 `recommendedExpression`，页面和 store 直接消费这个字段。

**下次复用方式**  
只要某个值需要长期稳定显示或回填，就把它当正式数据字段，不要当文案副产品。

### 4. 多端同步不能依赖整包覆盖

**问题**  
本地和 Vercel、云端之间的学习资产会越用越分叉，尤其是 review 状态和补全字段。

**为什么会发生**  
客户端拿自己的整包快照去抢写云端，最后一次写入覆盖了其他端的有效更新。

**有效修法**  
把 `/sync/save` 改成服务端 merge；客户端初始对齐也做 merge，而不是简单替换。

**下次复用方式**  
新增任何可持久化字段时，先思考它在多端更新时应该如何 merge，不要只考虑单端写入。

### 5. 词卡 review metadata 要和内容字段分开看

**问题**  
两端词卡总数相同，但“待复习数量”不同。

**为什么会发生**  
词卡内容没变，但 `reviewStage / lastViewedAt / nextDueAt` 已经变了；如果只按内容 `timestamp` merge，就会丢复习结果。

**有效修法**  
为 vocab cards 单独写 review winner 规则，而不是沿用通用 merge。

**下次复用方式**  
只要某个对象同时承载“内容”和“进度”，就要先判断这两类字段是否应该用不同的 merge 规则。

### 6. 长页面继续膨胀前先拆边界

**问题**  
页面越来越大，任何小改动都要重新读几百行，改一个点很容易误伤另一个区块。

**为什么会发生**  
状态、文案、交互、业务判断、展示结构都缠在同一个页面文件里。

**有效修法**  
把页面改成 route-level orchestration + 业务面板 / page controller。

**下次复用方式**  
如果一个页面已经承担多个明显独立任务块，就不要继续往里塞逻辑，先拆。

### 7. 学习资产必须有对称控制动作

**问题**  
对象可以新增，但删不掉、改不了、恢复不了，最后用户只能重建或绕路。

**为什么会发生**  
设计时只考虑了“生成”，没把这些对象当长期资产。

**有效修法**  
补齐删除、编辑、恢复等对称动作，并把它们做进 domain 层而不是只做 UI 假按钮。

**下次复用方式**  
新增任何长期对象前，先列出它至少需要的控制动作，而不是只实现“创建成功”。

## 反复出现的失败模式

### 1. 页面 effect 因自我更新而打断后续流程

典型现象是：第一条处理成功，后面的候选项全被取消，下一次打开又重新处理。  
这类问题不是“保存没生效”，而是页面补全过程被自己的 store 更新打断了。

复用修法：把候选项改成独立任务，不要把整组异步工作绑在一条会因为重渲染被 cleanup 的串行 effect 上。

### 2. UI 显示值和真实数据字段分离

典型现象是：页面看起来有内容，但真实对象里没有对应字段；刷新后就回到初始状态。  
复用修法：只要某个显示结果需要跨刷新保留，就必须让它进入正式模型字段。

### 3. 前端 parser 与 Edge Function 返回结构不同步

典型现象是：AI 实际返回了某些信息，但页面像空白或 fallback。  
复用修法：每次改后端输出时，同步看前端 parser、store 字段和消费组件。

## 不要再用的做法

- 不要把 AI 失败伪装成看似合理的通用建议
- 不要从长文案中正则猜标题、推荐表达或主展示值
- 不要把学习状态放在页面内临时对象里却不回写 store
- 不要让同步逻辑重新退化成整包覆盖
- 不要在高频页面继续堆大块业务逻辑而不拆边界

## 仍然有效的关键历史决策

- invite-only signup 是对的，不能回到公开 signup
- local-first 是对的，不能把核心练习链路绑成网络必达
- 错题正确句优先展示是对的，不要回到突出错误句
- 词卡的语体解析必须围绕“真实怎么说”，不要回到词典式解释
- 首页已经明确回避强制任务链，后续不要再把产品推回高压任务系统

如果未来又遇到类似问题，优先复用这里的修法模式，而不是重新发明一套新逻辑。
