# VerbaLab 关键边界

## 如何使用这份文档

这不是价值观陈述，而是一份“不能乱破坏的边界清单”。如果要修改产品行为、数据模型、同步逻辑或 AI 返回结构，先用这份文档判断自己是不是正在动红线。

## 产品层不变量

### 实验室不能退化成直接给答案

实验室的价值在于“先尝试输出，再获得支架和反馈”。它可以给提示、降难度、句块重排，但不能直接把完整标准答案作为主流程喂给用户，否则实验室就会退化成被动消费页面。

### 实战仓的降难度必须降低任务复杂度，而不是直接给整段回答

实战仓面对的是更长输出。支架可以是中文骨架、句首模板、关键句重排，但不应该直接提供整段可提交答案，否则用户只是照抄，不是在练持续表达。

### 词卡必须服务“真实怎么说”

词卡的重点不是定义，而是：

- 这个词在真实表达里是不是常用
- 什么时候该换说法
- 什么时候原词就能直接说

如果改动让词卡越来越像词典，而不是表达决策卡，就偏离产品定位了。

### 错题库必须强化正确句

错题库不能把错误句做成主视觉。正确句应该是主显示，错误句必须弱化。任何改动如果重新把错误句放成主角，都会伤害学习效果。

## 界面与学习体验红线

### 不要强迫用户按固定任务链走

这个产品已经明确偏向“把自由度还给用户”。首页、导航、练习区都不应强制规定今天必须先做什么、再做什么。推荐可以有，但不能做成高压任务系统。

### 卡壳点、词卡、错题都属于长期资产，不是一次性结果

任何相关页面都不能只考虑“当下显示对不对”，还必须考虑：

- 刷新后还在不在
- 重新进入页面是否还成立
- 是否可以编辑 / 删除 / 恢复

### 长句词块重排不能切得过散

无论是词卡复习还是实战仓关键句重排，切块都要有“可读性 + 进度感”。如果切得像一堆散词、用户不知道自己已完成多少，就会直接降低可用性。

## 数据与持久化红线

### 本地优先不能被网络依赖破坏

VerbaLab 的核心学习链路必须在本地可工作。网络请求可以增强 AI 质量，但不能让核心对象只活在远端响应里。

### 改持久化模型时必须同时更新 4 处

任何持久化对象 shape 变动，都必须至少同时检查：

- [types.ts](/Users/yuchao/Documents/GitHub/VerbaLab/src/app/store/types.ts)
- [persistence.ts](/Users/yuchao/Documents/GitHub/VerbaLab/src/app/store/persistence.ts)
- [useCloudSync.ts](/Users/yuchao/Documents/GitHub/VerbaLab/src/app/store/useCloudSync.ts)
- [sync.ts](/Users/yuchao/Documents/GitHub/VerbaLab/supabase/functions/make-server-1fc434d6/routes/sync.ts)

漏掉任意一处，都可能出现“页面看起来成功，但刷新后没了”或“本地和云端状态分叉”。

### 学习资产对象必须有对称控制动作

语料、错题、词卡、卡壳点这类长期对象，不应该只支持“新增”。如果对象会长期存在，就通常需要成对考虑：

- 删除
- 编辑
- 标记已解决
- 恢复为未解决

缺任何一个，都容易演变成用户只能绕路重建。

## AI 返回契约红线

### 关键字段必须结构化返回

如果某个 UI 需要稳定显示一个核心值，例如：

- 卡壳点推荐表达
- 词卡语体解析
- 错题正确句

就应该让 AI 明确返回字段，而不是从一大段文本里反向提取。猜字段是高频故障源。

### 改 AI response shape 时必须前后端一起改

修改 AI 输出时，至少同步检查：

- [api.ts](/Users/yuchao/Documents/GitHub/VerbaLab/src/app/utils/api.ts)
- 调用它的页面 / 组件
- 对应的 Edge Function route

如果后端字段变了，前端 parser 不变，就会出现“AI 实际返回了，但 UI 看起来像没返回”。

### 不要用静默 fallback 伪装 AI 成功

如果 AI 请求失败，而页面却展示一个看似合理、其实与用户输入无关的本地兜底内容，用户会被误导，系统也会失去可诊断性。失败时就应该明确暴露失败，而不是伪造成功。

## 同步与多端状态红线

### 不要回到整包覆盖

当前同步已经从“谁最后写谁覆盖”往 merge 方向收敛。对 corpus、error bank、stuck points、vocab cards 的处理都带有合并逻辑。任何新改动都不要把它重新拉回简单覆盖。

### 词卡 review metadata 不能按普通内容对待

词卡的 `reviewStage / lastViewedAt / nextDueAt` 与一般内容字段不是同一种东西。改动词卡同步时，必须单独考虑 review metadata merge，而不是只比 `timestamp`。

### 删除传播是已知薄弱边界

当前删除还没有 tombstone 语义。涉及多端删除一致性时，不能假设“云端没有这条就是删除了”。这块在 [sync-data-flow.md](/Users/yuchao/Documents/GitHub/VerbaLab/docs/sync-data-flow.md) 已经明确写明。

## 部署与线上风险边线

### 改跨域行为必须检查 CORS

Vercel 域名、preview 域名、自定义域名都可能影响 Edge Function 请求。只要动到线上调用链、域名或新的调用入口，就要检查 CORS 规则和 Supabase secrets。

### 改注册链路不能只改前端

邀请码注册真正的安全边界不在 UI，而在：

- 关闭公开 signup
- 受控 Edge Function 注册入口
- invites 表消费逻辑

只改前端表单是没有意义的。

### 验证是改动的一部分

任何有行为影响的改动，至少要跑 `npm run typecheck`。逻辑改动要跑 `npm run test`，边界或打包改动要跑 `npm run build`。不验证就宣称完成，会把未来调试成本全部转嫁给下一位接手者。

## 什么情况下必须停下来重新检查

出现下面这些迹象时，不要继续顺手修，先回头检查边界：

- 页面显示结果对，但刷新后消失
- 本地和云端数量一致，但 due / resolved / corrected state 不一致
- AI 明明调用了，却展示与输入无关的内容
- 一处小交互改动，却牵动大量不相干状态
- 需要从长文案里“猜”某个核心展示字段

这些通常不是单点 bug，而是数据、契约或页面边界出了问题。
